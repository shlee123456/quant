# LLM + VectorBT 통합 아키텍처 설계 문서

이 문서는 VectorBT 마이그레이션 및 LLM 통합 파이프라인의 설계/개발 기준 문서입니다.

---

## 1. 현재 백테스팅 구조 (자체 구현)

### 자체 구현 장점
- 프로젝트 요구사항에 정확히 맞는 커스터마이징 가능
- 외부 의존성 최소화 (버전 충돌/deprecation 걱정 없음)
- 내부 로직을 완전히 이해하고 제어 가능
- 가볍고 빠름 (필요한 기능만 구현)
- 디버깅이 쉬움

### 자체 구현 단점
- 직접 검증해야 함 (버그 가능성)
- 고급 기능 직접 구현 필요 (슬리피지, 부분 체결, 멀티 에셋 포트폴리오 등)
- 개발/유지보수 비용
- 엣지 케이스 누락 가능성 (look-ahead bias 방지 등)

### 주요 백테스팅 라이브러리 비교

| 라이브러리 | 특징 |
|-----------|------|
| **Backtrader** | 이벤트 기반, 풍부한 기능, 커뮤니티 큼. 다만 유지보수 둔화 |
| **Zipline** | Quantopian 출신, 분/일봉 지원, pandas 통합. 설치 까다로움 |
| **VectorBT** | 벡터화 연산으로 매우 빠름, 대규모 파라미터 최적화에 강점 |
| **bt** | 포트폴리오 레벨 백테스트, 자산 배분 전략에 적합 |

### VectorBT 선정 이유
- 벡터화 연산으로 수백~수천 배 빠른 파라미터 최적화
- Phase 4(ML 전략, 포트폴리오 최적화)에서 대규모 파라미터 탐색 필수
- LLM 판단의 근거가 될 레짐별 전략 성과 데이터를 빠르게 생성 가능

## 1.1 현재 상황 진단 (2026-02-13)

### 핵심 문제: 시뮬레이션 데이터 최적화 → 실제 시장에서 거래 0건

Phase 2(스케줄러 자동화)까지 완료되었으나, **자동 최적화된 파라미터로 실제 시장에서 거래가 단 한 건도 발생하지 않는 문제**가 확인됨.

#### 원인 분석

`scheduler.py`의 `optimize_strategy()` 함수가 `SimulationDataGenerator(GBM)`으로 생성한 가상 데이터를 사용하여 최적화를 수행. 이 데이터는 실제 시장과 특성이 크게 다름.

```
시뮬레이션 데이터 (GBM)              실제 시장 데이터
━━━━━━━━━━━━━━━━━━━━━━              ━━━━━━━━━━━━━━━
변동성: 높음 (2~3배)                 변동성: 낮음
추세: 명확 (상승/하락/횡보)            추세: 무작위 보행 특성
가격 범위: ~127%                    가격 범위: ~26%
최적화 결과: 8~10회 거래 발생          실전 결과: 0회 거래 발생
```

RSI_MACD_Combo 전략의 최적화 결과 파라미터 `RSI 38/75`에서:
- BUY 조건 (`RSI < 38 AND MACD 골든크로스`) 동시 충족 확률: **~1%**
- 7.5시간 운영 결과: 신호 1,915개 중 **BUY 0개, SELL 0개, 전부 HOLD**

이는 **과적합(Overfitting)**의 전형적 사례: 학습 데이터(시뮬레이션)와 테스트 데이터(실제 시장)의 분포가 다름.

#### 해결 방안

| 단계 | 작업 | 상태 |
|------|------|------|
| **0단계** | 실제 시장 데이터 기반 최적화로 전환 (`scheduler.py` 수정) | ✅ 완료 |
| **1단계** | VBT 마이그레이션 (대규모 파라미터 탐색 속도 개선) | ✅ 완료 (2026-02-13) |
| **2단계** | 레짐 감지 모듈 (시장 상태별 전략 자동 전환) | ✅ 완료 (2026-02-13) |
| **3단계** | LLM 시그널 필터 + 레짐 판단 연동 | ✅ 완료 (2026-02-13) |

#### LLM 연동 시작 조건 (체크리스트)

- [ ] 최소 30일 이상의 Paper Trading 세션 데이터
- [ ] 전략별 100회 이상의 거래 기록
- [ ] 레짐별(상승/하락/횡보) 전략 성과 비교 데이터
- [ ] VBT 기반 백테스트로 파라미터 민감도 분석 완료

**원칙**: LLM은 데이터를 생성하지 않고 필터링한다. 필터링할 데이터가 없으면 LLM은 무의미.

## 2. VectorBT 마이그레이션 작업

### 2.1 시그널 생성 방식 변경 (전략 레이어)

현재 전략들은 DataFrame에 `signal` 컬럼을 만드는 방식. VectorBT는 entries/exits 불리언 배열 방식.

```python
# 현재 (자체 구현)
data['signal'] = 0
data.loc[rsi < 30, 'signal'] = 1   # BUY
data.loc[rsi > 70, 'signal'] = -1  # SELL

# VectorBT 방식
entries = rsi_crossed_below(30)  # Boolean array
exits = rsi_crossed_above(70)    # Boolean array
pf = vbt.Portfolio.from_signals(close, entries, exits)
```

영향 파일: `strategies/` 하위 5개 전략 전부

### 2.2 Backtester 교체 (핵심)

`trading_bot/backtester.py`의 `run()` 루프가 `vbt.Portfolio.from_signals()`로 대체.

```python
# 현재: Python for-loop로 바별 순회
for i in range(len(data)):
    if signal == 1 and position == 0: ...

# VectorBT: 벡터화된 단일 호출
pf = vbt.Portfolio.from_signals(
    close=df['close'],
    entries=entries,
    exits=exits,
    init_cash=10000,
    size=0.95,
    fees=0.001
)
results = pf.stats()
```

### 2.3 Optimizer 리팩토링 (가장 큰 이점)

현재 `StrategyOptimizer`는 파라미터 조합마다 Backtester를 새로 만들어 순차 실행. VectorBT는 파라미터 조합을 한번에 벡터화.

```python
# 현재: N번 반복 (느림)
for params in itertools.product(...):
    backtester = Backtester(strategy_class(**params))
    backtester.run(df)

# VectorBT: 한번에 실행 (수백~수천 배 빠름)
rsi = vbt.RSI.run(close, window=[10, 14, 20])
entries = rsi.rsi_crossed_below([20, 25, 30])
pf = vbt.Portfolio.from_signals(close, entries, exits)
pf.total_return()  # 모든 조합 결과 한번에
```

### 2.4 메트릭 매핑

| 현재 자체 계산 | VectorBT 대응 |
|---------------|---------------|
| `total_return` | `pf.total_return()` |
| `sharpe_ratio` | `pf.sharpe_ratio()` |
| `max_drawdown` | `pf.max_drawdown()` |
| `win_rate` | `pf.trades.win_rate()` |
| `equity_curve` | `pf.value()` |
| `trades` | `pf.trades.records_readable` |

반환 형식이 다르므로 기존 Dashboard/PaperTrader가 기대하는 Dict 형식으로 변환하는 어댑터 필요.

### 2.5 PaperTrader는 마이그레이션 대상 아님

PaperTrader는 실시간 루프로 바별 처리가 필수이므로 VectorBT 대체 불가. 기존 로직 유지하되 전략 인터페이스만 호환.

### 2.6 작업 매트릭스

| 작업 | 난이도 | 영향 범위 |
|------|--------|----------|
| VectorBT 의존성 추가 | 낮음 | `requirements.txt` |
| BaseStrategy에 entries/exits 메서드 추가 | 중간 | `strategies/` 전체 |
| Backtester를 VBT wrapper로 교체 | 중간 | `backtester.py` |
| Optimizer 벡터화 리팩토링 | 높음 | `optimizer.py` |
| 결과 어댑터 (Dict 호환 레이어) | 중간 | 신규 |
| Dashboard 연동 수정 | 중간 | `dashboard/tabs/` |
| 기존 테스트 수정 | 중간 | `tests/` 전체 |
| PaperTrader 전략 호환 | 낮음 | `paper_trader.py` |

### 2.7 권장 접근법: 점진적 마이그레이션

1. BaseStrategy에 `get_entries_exits()` 메서드 추가 (기존 `calculate_indicators`와 공존)
2. `VBTBacktester` 래퍼 클래스를 새로 만들어 기존 `Backtester`와 병행
3. Optimizer에서 VBT 경로 옵션 추가
4. 검증 후 기존 코드 제거

### 3.2 LLM의 3가지 역할

#### 역할 1: 시그널 필터 (핵심)

VBT가 생성한 기술적 시그널을 실행할지 말지 판단.

```
VBT: "RSI 28 → BUY 시그널"
LLM: "현재 FOMC 발표 2시간 전 + 뉴스 센티먼트 극도 부정적
      → 시그널 보류, 발표 후 재평가"
```

LLM은 시그널을 만들지 않고, 퀀트 엔진이 놓치는 정성적 맥락을 보완.

#### 역할 2: 전략 선택자 (레짐 기반)

마켓 레짐에 따라 어떤 전략을 쓸지 결정.

```
LLM 판단:
  레짐: 강한 상승 추세 → MA Crossover 활성화
  레짐: 횡보 구간     → RSI 역추세 전략 활성화
  레짐: 급변/위기     → 전략 비활성화, 현금 보유
```

VBT의 과거 백테스트 결과를 레짐별로 분류한 데이터가 근거.

#### 역할 3: 리스크 조절자

확신도에 따라 포지션 크기를 동적으로 조정.

```
높은 확신 (VBT 시그널 + 센티먼트 일치 + 유리한 레짐) → position_size: 0.95
중간 확신 (시그널은 있으나 이벤트 리스크 존재)       → position_size: 0.50
낮은 확신 (시그널과 맥락 불일치)                    → position_size: 0 (보류)
```

### 3.3 VBT와 LLM의 분업

```
VBT (빠르고 정량적)              LLM (느리고 정성적)
━━━━━━━━━━━━━━━━━━              ━━━━━━━━━━━━━━━━━━
• 수천 개 파라미터 최적화          • 뉴스/이벤트 해석
• 밀리초 단위 백테스트             • "왜" 에 대한 판단
• 통계적 엣지 발견                • 과거에 없던 상황 대응
• entries/exits 생성             • 시그널 필터링/크기 조정
• 과적합 감지 (WFO)              • 자연어 리포트 생성
```

원칙: **VBT가 후보를 만들고, LLM이 최종 판단한다.**

LLM이 직접 매매 시그널을 만들면 할루시네이션 리스크가 있으므로, 퀀트 엔진의 시그널을 검증/보완하는 역할로 제한.

## 4. LLM 모델 선정

### 4.1 역할별 요구사항

| 역할 | 응답 시간 | 호출 빈도 | 요구 능력 |
|------|----------|----------|----------|
| 뉴스 센티먼트 | 배치 (느려도 됨) | 수백건/일 | 분류 정확도 |
| 시그널 필터 | < 2초 | 분당 1회 | 추론 + 구조화 출력 |
| 레짐 판단 | < 10초 | 1-2회/일 | 강한 추론 |
| 리포트 생성 | 배치 | 1회/일 | 문장력 |
| 피드백 자기평가 | 배치 | 1회/주 | 강한 추론 + 장문맥 |

### 4.2 역할별 모델 추천

#### 뉴스 센티먼트 분석 (분류 태스크 → 경량 특화 모델)

| 모델 | 파라미터 | 비고 |
|------|---------|------|
| **FinBERT** | **110M** | 금융 특화 BERT, 가장 실용적 |
| ProsusAI/finbert | 110M | HuggingFace에서 바로 사용 |
| Fine-tuned Qwen2.5 | 0.5B-1.5B | 더 복잡한 맥락 필요 시 |

- GPU 불필요 (CPU 추론 가능, ~10ms/건)
- 수백 건의 뉴스를 배치로 처리해도 수 초
- 110M이면 충분, LLM 쓸 이유 없음

#### 시그널 필터 (실시간 판단 → 소형 LLM)

| 방식 | 모델 | 파라미터 | 응답시간 | 비용 |
|------|------|---------|---------|------|
| **로컬 (권장)** | **Qwen2.5-Coder 7B** | **7B** | ~0.5초 | 전기세만 |
| 로컬 | Llama 3.1 8B | 8B | ~0.5초 | 전기세만 |
| 로컬 | Mistral 7B | 7B | ~0.4초 | 전기세만 |
| API 대안 | Claude Haiku 4.5 | - | ~0.8초 | ~$0.001/호출 |
| API 대안 | GPT-4o-mini | - | ~0.7초 | ~$0.001/호출 |

7B 추천 이유:
- 입력이 구조화되어 있고 출력도 JSON 고정 → 패턴 매칭에 가까움
- 7B면 4bit 양자화 시 VRAM 4-6GB → RTX 3060급이면 충분
- 70B 수준의 추론력이 필요한 태스크가 아님

#### 레짐 판단 + 전략 선택 (복합 추론 → 중형 모델)

| 방식 | 모델 | 파라미터 | 비고 |
|------|------|---------|------|
| **로컬 (권장)** | **Qwen2.5 14B** | **14B** | 추론력 대비 크기 최적 |
| 로컬 | Llama 3.1 70B | 70B | 최고 품질, GPU 요구 높음 |
| API 대안 | Claude Sonnet 4.5 | - | 가장 안정적 |

- 하루 1-2회 호출이므로 응답시간 10초도 무방
- 14B → 4bit 양자화 시 VRAM 10-12GB → RTX 4070급
- 장문의 백테스트 결과를 맥락으로 넣어야 하므로 14B 이상 권장

#### 리포트 생성 + 피드백 (장문 생성 → API)

| 방식 | 모델 | 비고 |
|------|------|------|
| **API (권장)** | **Claude Sonnet 4.5** | 비용 대비 품질 최고 |
| API | GPT-4o | 대안 |
| 로컬 | Qwen2.5 32B | 오프라인 필요 시 |

- 하루/주 1회 배치이므로 API 비용 미미 (~$0.05/리포트)
- 긴 거래 이력을 분석해야 하므로 큰 컨텍스트 윈도우 필요

### 4.3 권장 배포 구성

```
┌─────────────────────────────────────────────────────┐
│              실전 배포 구성                           │
├─────────────────────────────────────────────────────┤
│                                                     │
│  로컬 GPU (RTX 4070 12GB 1장)                       │
│  ┌───────────────────────────────────────┐          │
│  │ FinBERT 110M      ← 센티먼트 (CPU)    │          │
│  │ Qwen2.5 7B (4bit) ← 시그널 필터 (GPU) │          │
│  │ Qwen2.5 14B (4bit)← 레짐 판단 (GPU)   │          │
│  └───────────────────────────────────────┘          │
│  * 7B와 14B는 동시 로딩 X, 시간대별 교체 로딩         │
│  * 7B: 장중 상주 / 14B: 장 시작 전 1회 실행          │
│                                                     │
│  API (배치 작업)                                     │
│  ┌───────────────────────────────────────┐          │
│  │ Claude Sonnet 4.5  ← 일일 리포트       │          │
│  │ Claude Sonnet 4.5  ← 주간 피드백       │          │
│  └───────────────────────────────────────┘          │
│  * 월 API 비용: ~$5-15                              │
│                                                     │
│  총 파라미터: 110M + 7B + 14B + API                  │
│  필요 VRAM: 12GB (모델 교체 로딩 방식)                │
│  필요 RAM: 16GB+                                    │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 4.4 거대 모델(70B+)이 불필요한 이유

| 이유 | 설명 |
|------|------|
| 입력이 구조화됨 | 자유 텍스트가 아니라 수치+라벨 조합 → 7B로 충분 |
| 판단 패턴이 유한 | BUY/HOLD/SELL x 3-5개 맥락 요인 → 복잡한 추론 불필요 |
| 레이턴시 | 70B는 응답 3-5초, 시그널 필터에는 과함 |
| 비용 | 70B 로컬 = A100급 GPU, API = 호출당 10배 비용 |
| 파인튜닝 | 7B는 LoRA로 1시간 이내 파인튜닝 가능, 70B는 수일 |

## 5. 구현 우선순위

| 순서 | 작업 | 이유 |
|------|------|------|
| 1 | VBT 마이그레이션 | LLM 판단의 근거가 될 빠른 백테스트 인프라 |
| 2 | 레짐 감지 모듈 | 전략 선택의 기반, LLM 없이도 동작 가능 |
| 3 | 뉴스 센티먼트 파이프라인 | LLM 입력 데이터 확보 |
| 4 | LLM 메타 판단 엔진 | 시그널 필터 + 포지션 크기 조정 |
| 5 | 피드백 루프 | 판단 결과를 학습에 반영 |

VBT가 없으면 LLM에게 줄 데이터(과거 레짐별 전략 성과, 파라미터 민감도 등)를 빠르게 생성할 수 없으므로 VBT가 선행되어야 한다.
